# Phase 13: Design System Foundation - Research

**Researched:** 2026-03-21
**Domain:** Design tokens, cross-platform theming, monorepo package structure
**Confidence:** HIGH

## Summary

Phase 13 creates a shared design token package at `packages/shared/tokens/` that becomes the single source of visual truth for both mobile (React Native) and web (Next.js + Tailwind v4). The existing mobile `theme.ts` (13 colors, 5 typography styles, 6 spacing sizes, 5 border radius levels) and web `constants.ts` THEME object (8 color values) get unified into a two-tier token hierarchy: primitive values and semantic aliases.

The migration strategy is conservative and well-scoped. Mobile gets a backward-compatible shim (re-exporting new tokens through old property names) so the 44 consuming files need zero changes. Web has only 2 files with hardcoded hex values (constants.ts + globals.css), making direct replacement safe. A Node build script generates `tokens.css` from the TS token source for Tailwind v4's CSS-first @theme configuration.

The codebase audit identified 149 hardcoded hex occurrences across 29 mobile files (beyond theme.ts itself), plus type-specific colors for Pokemon TCG rarities (diamond blue #7ec8e3, star gold #f0c040, crown purple #e8b4f8), tier list grades (S through D), and energy types (10 colors). These are documented but NOT migrated in this phase -- Phases 16/17 handle file-by-file migration.

**Primary recommendation:** Build the token package as plain TypeScript objects with zero runtime dependencies, generate CSS via a simple Node script in the Turborepo pipeline, and keep the shim layer as a pure re-export mapping.

### Pattern 2: Backward-Compatible Shim
**What:** Old `theme.ts` becomes a thin re-export layer mapping old property names to new tokens.
**When to use:** For mobile migration -- preserves the existing import API.
**Example:**
```typescript
// apps/mobile/src/constants/theme.ts (AFTER migration)
import { colors as tokenColors, spacing as tokenSpacing, typography as tokenTypography, borderRadius as tokenBorderRadius } from '@pocket-trade-hub/shared/tokens';

// Re-export with exact same shape as before
export const colors = {
  background: tokenColors.background,
  surface: tokenColors.surface,
  surfaceLight: tokenColors.surfaceLight,
  primary: tokenColors.accent,          // old name -> new semantic
  primaryDark: tokenColors.accentDark,  // old name -> new semantic
  text: tokenColors.onSurface,
  textSecondary: tokenColors.onSurfaceSecondary,
  textMuted: tokenColors.onSurfaceMuted,
  error: tokenColors.error,
  success: tokenColors.success,
  border: tokenColors.border,
  tabBar: tokenColors.tabBar,
  inputBackground: tokenColors.inputBackground,
};

export { tokenSpacing as spacing };
export { tokenBorderRadius as borderRadius };
export { tokenTypography as typography };
```

### Pattern 3: CSS Token Generation
**What:** Node script reads TS token objects and writes CSS custom properties file.
**When to use:** For web platform -- bridges TS tokens to Tailwind v4 @theme.
**Example:**
```typescript
// packages/shared/scripts/generate-css-tokens.ts
import { colors } from '../src/tokens/colors';
import { spacing } from '../src/tokens/spacing';

function camelToKebab(str: string): string {
  return str.replace(/([a-z])([A-Z])/g, '$1-$2').toLowerCase();
}

function generateCSS(): string {
  const lines: string[] = ['/* AUTO-GENERATED -- do not edit manually */', '@theme {'];

  // Colors
  for (const [key, value] of Object.entries(colors)) {
    lines.push(`  --color-${camelToKebab(key)}: ${value};`);
  }

  // Spacing (convert to rem for web)
  for (const [key, value] of Object.entries(spacing)) {
    lines.push(`  --spacing-${key}: ${value / 16}rem;`);
  }

  lines.push('}');
  return lines.join('\n');
}

// Write output
import { writeFileSync } from 'fs';
import { resolve } from 'path';
const outPath = resolve(__dirname, '../../../apps/web/src/app/tokens.css');
writeFileSync(outPath, generateCSS());
console.log(`Generated ${outPath}`);
```

### Anti-Patterns to Avoid
- **Importing primitives directly in components:** Components should ONLY use semantic tokens. Primitives are internal to the token package.
- **Adding component-level tokens in Phase 13:** Decision D-04 explicitly defers this. Keep it to primitive + semantic only.
- **Changing visual output during migration:** The shim must produce IDENTICAL values to the old theme.ts. Any palette changes are separate from the shim.
- **Generating tokens at runtime:** The CSS generation is a build step, not a runtime operation.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Token type safety | Manual type declarations | `as const` + `typeof` inference | TypeScript infers exact types from const objects automatically |
| CSS variable naming | Manual string concatenation | camelToKebab utility function | Consistent, testable conversion |
| Audit tracking | Manual spreadsheet | Script-generated JSON/markdown | Reproducible, version-controlled, can be re-run |

**Key insight:** The token package itself IS the build tool -- plain TS objects with a simple Node script. No framework needed.

## Common Pitfalls

### Pitfall 1: Shim API Mismatch
**What goes wrong:** The shim changes property names or shapes, breaking downstream imports silently (especially destructured imports like `const { background } = colors`).
**Why it happens:** Renaming during migration without checking all import patterns.
**How to avoid:** Map the exact current API of theme.ts first, then ensure the shim exports the identical shape. Run TypeScript compilation across the mobile app after shim swap.
**Warning signs:** TypeScript errors in any of the 44 theme-importing files.

### Pitfall 2: Palette Drift Between Platforms
**What goes wrong:** Mobile and web show visually different colors despite "shared" tokens.
**Why it happens:** Mobile uses raw hex values; web may apply opacity or color-mix via Tailwind utilities. Also, the current palettes are already different (mobile #0f0f1a vs web #0a0a0a).
**How to avoid:** The unified palette (D-01) creates ONE source. Verify both platforms render the same hex values.
**Warning signs:** Side-by-side comparison shows mismatched backgrounds or surfaces.

### Pitfall 3: Circular Imports in Shared Package
**What goes wrong:** Adding tokens to the shared package index creates circular dependency if tokens import from schemas or vice versa.
**Why it happens:** Barrel re-exports in index.ts can create unexpected dependency chains.
**How to avoid:** Keep tokens completely isolated -- `tokens/index.ts` exports only token values, no schema imports. Re-export from shared `index.ts` as a separate export path.
**Warning signs:** Runtime "Cannot access before initialization" errors.

### Pitfall 4: Generated CSS File in Git
**What goes wrong:** `tokens.css` is generated but gets committed, leading to merge conflicts and stale values.
**Why it happens:** Forgetting to add to .gitignore.
**How to avoid:** Add `apps/web/src/app/tokens.css` to .gitignore. Generate it in the build pipeline.
**Warning signs:** Git diff shows tokens.css changes on every commit.

### Pitfall 5: Spacing Unit Mismatch (px vs rem)
**What goes wrong:** Mobile spacing is in pixels (4, 8, 16...) but web should use rem for accessibility.
**Why it happens:** Token package exports raw numbers; web CSS generation needs to convert.
**How to avoid:** Token package exports raw numbers (platform-agnostic). CSS generation script divides by 16 for rem. Mobile uses raw numbers directly in StyleSheet.
**Warning signs:** Web spacing looks wrong or doesn't scale with browser font size.

### Pitfall 6: Warning Color Too Close to Gold Accent
**What goes wrong:** Users cannot distinguish warnings from accent/primary elements.
**Why it happens:** Orange and gold are adjacent on the color wheel.
**How to avoid:** Use a distinctly orange hue (#e67e22 range) rather than amber/yellow. Test by displaying warning toast next to gold-accented UI.
**Warning signs:** Feedback that "everything looks gold."

## Open Questions

1. **Exact unified dark palette values**
   - What we know: Current mobile uses #0f0f1a background with blue undertone; web uses #0a0a0a (neutral dark). Gold accent #f0c040 is shared.
   - What's unclear: The exact gray scale values for the unified palette.
   - Recommendation: Claude's discretion per CONTEXT.md. Use blue-undertone darks (closer to mobile's character) since this is primarily a mobile app. Proposed scale in the primitives example above.

2. **Shared package export path**
   - What we know: Both apps import from `@pocket-trade-hub/shared`. Current barrel export is `src/index.ts`.
   - What's unclear: Whether to use a sub-path export (`@pocket-trade-hub/shared/tokens`) or nest in the main barrel.
   - Recommendation: Add to main barrel export (`export * from './tokens'`) for simplicity. Sub-path exports require package.json `exports` field configuration which adds complexity for no real benefit at this scale.

3. **Energy type colors (10 values in CardThumbnail)**
   - What we know: 10 Pokemon energy type colors are hardcoded in CardThumbnail.tsx (Fire, Water, Grass, etc.).
   - What's unclear: Whether these should be first-class tokens or remain as a domain-specific constant.
   - Recommendation: Include as `palette.energy.*` primitives since they appear in multiple components and represent a fixed, well-known set. But do NOT add semantic aliases -- these are domain constants, not UI semantics.
