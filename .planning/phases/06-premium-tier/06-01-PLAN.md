  <behavior>
    - setPremiumStatus(db, userId, true, expiresAt) sets isPremium=true and premiumExpiresAt on user
    - setPremiumStatus(db, userId, false, null) sets isPremium=false and clears premiumExpiresAt
    - handleWebhookEvent processes INITIAL_PURCHASE/RENEWAL/UNCANCELLATION -> premium=true
    - handleWebhookEvent processes EXPIRATION -> premium=false
    - handleWebhookEvent on CANCELLATION does NOT revoke (user paid through period)
    - isPremiumUser returns true when isPremium=true, false otherwise
    - computeAllAnalytics populates cardAnalytics table with most_wanted, least_available, trending metrics
    - getAnalytics returns top-10 per metric from pre-computed cardAnalytics table
    - getTradePower returns cards from user's collection that are most wanted by others
    - checkCardAlerts creates cardAlertEvents for premium users who want the added card (excludes adder)
    - processCardAlertBatch groups pending events by user, creates batched notification, marks processed
  </behavior>
  <action>
    1. Create `packages/shared/src/schemas/premium.ts`:
       - Zod schemas: `analyticsCardSchema` (cardId, cardName, cardImageUrl, rarity, value, rank), `analyticsResponseSchema` (mostWanted, leastAvailable, trending, tradePower arrays), `subscriptionStatusSchema` (isPremium, premiumExpiresAt nullable)
       - Export inferred TypeScript types

    2. Update `apps/api/src/db/schema.ts`:
       - Add to users table: `isPremium: boolean('is_premium').default(false).notNull()`, `premiumExpiresAt: timestamp('premium_expires_at')`, `revenuecatId: varchar('revenuecat_id', { length: 100 })`
       - Add `cardAnalytics` table: id (text PK), cardId (text FK->cards), metric (varchar 30), value (integer), rank (integer), computedAt (timestamp). Indexes: (metric, rank), unique (cardId, metric)
       - Add `cardAlertEvents` table: id (text PK), premiumUserId (text FK->users), cardId (text FK->cards), addedByUserId (text FK->users), processed (boolean default false), createdAt (timestamp). Index: (premiumUserId, processed)

    3. Create `apps/api/src/services/premium.service.ts`:
       - `setPremiumStatus(db, userId, isPremium, expiresAt)`: Updates users table isPremium + premiumExpiresAt
       - `isPremiumUser(db, userId)`: Returns boolean from users table
       - `handleWebhookEvent(db, event)`: Switch on event.type -- INITIAL_PURCHASE/RENEWAL/UNCANCELLATION -> setPremiumStatus(true), EXPIRATION -> setPremiumStatus(false), CANCELLATION -> no-op (user paid through period)
       - Use `event.app_user_id` to find user, `event.expiration_at_ms` for expiry

    4. Create `apps/api/src/services/analytics.service.ts`:
       - `computeMostWanted(db)`: SQL count(*) on userWantedCards grouped by cardId, ordered desc, limit 10. Delete old most_wanted rows from cardAnalytics, insert new with rank 1-10
       - `computeLeastAvailable(db)`: Cards with fewest collection entries. Count userCollectionItems grouped by cardId from cards table (LEFT JOIN so cards with 0 owners appear). Order ASC, limit 10. Insert as least_available metric
       - `computeTrending(db)`: Cards with biggest increase in wanted-list additions over past 7 days. Count userWantedCards WHERE createdAt > 7 days ago, grouped by cardId, ordered desc, limit 10. Insert as trending metric
       - `computeAllAnalytics(db)`: Calls all three compute functions sequentially
       - `getAnalytics(db)`: Reads pre-computed rows from cardAnalytics, joins with cards table for name/imageUrl/rarity, groups by metric
       - `getTradePower(db, userId)`: Cards in user's collection that appear most in other users' wanted lists. Join userCollectionItems(userId) with userWantedCards(cardId, userId!=user), count, order desc, limit 10

    5. Create `apps/api/src/services/card-alert.service.ts`:
       - `checkCardAlerts(db, addedByUserId, cardId)`: Find premium users who want this cardId (join userWantedCards with users where isPremium=true, exclude addedByUserId). Insert cardAlertEvent row for each
       - `processCardAlertBatch(db)`: Select unprocessed cardAlertEvents grouped by premiumUserId. For each user: count unique cards, create a batched notification via createNotification ("3 traders added cards you want"), send push notification, mark events as processed. Use existing dual-delivery pattern from notification.service.ts

    6. Write tests (mock DB with in-memory patterns matching existing test files):
       - premium.service.test.ts: Test all event types, verify setPremiumStatus calls
       - analytics.service.test.ts: Test compute functions return correct rankings
       - card-alert.service.test.ts: Test alert creation excludes adder, test batch processing groups correctly

    Run `npx drizzle-kit push` after schema changes to apply to DB.
