  <action>
    1. **Install socket.io-client**: `cd apps/mobile && npx expo install socket.io-client`

    2. **Create `apps/mobile/src/stores/trades.ts`**: Zustand store (no persist -- matches are server-sourced):
       - State: `matches: TradeMatch[]`, `unseenCount: number`, `sortBy: MatchSort` (default 'priority'), `isLoading: boolean`, `lastRefreshed: number | null`
       - Actions:
         - `setMatches(matches, unseenCount)`: replace matches array and unseen count
         - `addMatch(match)`: prepend new match to array, increment unseenCount (for real-time Socket.IO events)
         - `clearUnseen()`: set unseenCount to 0 (called when Trades tab focused)
         - `setSortBy(sort)`: update sort preference
         - `setLoading(loading)`: toggle loading state
         - `markSeen(matchId)`: update match.seen=true in local array, decrement unseenCount

    3. **Create `apps/mobile/src/hooks/useMatches.ts`**: Data hook:
       - `useMatches()` returns `{ matches, unseenCount, isLoading, refresh, markSeen, sortBy, setSortBy }`
       - `loadMatches(sort?)`: GET /matches?sort={sortBy} via apiFetch, parse response, call store.setMatches
       - `refresh()`: POST /matches/refresh via apiFetch, then call loadMatches (immediate recompute + reload)
       - `markSeen(matchId)`: PUT /matches/{id}/seen via apiFetch, call store.markSeen
       - Call loadMatches on mount (useEffect with empty deps)
       - Re-fetch when sortBy changes (useEffect watching sortBy)

    4. **Create `apps/mobile/src/hooks/useMatchSocket.ts`**: Socket.IO connection hook:
       - Import `io` from `socket.io-client`
       - Get userId from useAuthStore
       - Get API_URL from environment/constants (same base URL as apiFetch)
       - In useEffect: if userId exists, create socket connection with `auth: { userId }` and `transports: ['websocket']`
       - Listen for `'new-match'` event:
         - Call `useTradesStore.getState().addMatch(matchData)` to update store
         - Show toast via `Toast.show({ type: 'matchNotification', text1: 'New match found!', text2: '${match.partnerName} has ${match.topCardName} you want.', visibilityTime: 5000, onPress: () => { /* navigation handled in Task 2 */ } })`
       - Cleanup: disconnect socket on unmount or userId change
       - Handle reconnection: socket.io-client handles this automatically
       - On logout (userId becomes null): disconnect socket

    5. **Update `apps/mobile/src/hooks/useCollection.ts`**: After successful collection mutations (addToCollection, removeFromCollection, bulkUpdateCollection), add a fire-and-forget call to `apiFetch('/matches/refresh', { method: 'POST' }).catch(() => {})`. This is non-blocking -- the user doesn't wait for match recompute. Only add the trigger after mutations that change the user's collection state (add, remove, bulk), not for read operations.

    6. **Update `apps/mobile/src/hooks/useWanted.ts`**: Same pattern -- after successful wanted list mutations (addToWanted, removeFromWanted, updatePriority), fire-and-forget POST /matches/refresh. Non-blocking.

    Note on mutation triggers: The POST /matches/refresh call from mobile is a convenience trigger. The backend BullMQ debounce handles deduplication, so rapid mutations from bulk-add won't cause N recomputes. The mobile just signals "something changed."
