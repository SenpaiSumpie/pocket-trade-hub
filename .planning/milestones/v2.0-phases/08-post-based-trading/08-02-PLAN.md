  <behavior>
    - findComplementaryMatches for an Offering post returns active Seeking posts with matching cardId+language from other users
    - findComplementaryMatches for a Seeking post returns active Offering posts with matching cardId+language from other users
    - findComplementaryMatches also returns wanted list matches for proactive notifications (Offering post matches someone's wanted list even without a Seeking post)
    - Notifications are batched per user: one notification per user even if multiple posts match
    - No self-matching (user's own posts/wanted list excluded)
    - Worker processes jobs with 5-second delay for batching rapid post creation
    - Socket.IO events emitted for post matches
  </behavior>
  <action>
    1. Create post-match.service.ts with:
       - findComplementaryMatches(db, post): Given a new post, find complementary active posts. For an Offering post: query tradePosts WHERE type='seeking' AND status='active' AND cards @> jsonb containment for matching cardId+language AND userId != post.userId. For a Seeking post: query tradePosts WHERE type='offering' AND status='active' with same containment logic.
       - findWantedListMatches(db, post): For Offering posts only, query userWantedCards WHERE cardId matches post card's cardId AND language matches AND userId != post.userId. This enables proactive notifications even without a Seeking post.
       - processPostMatch(db, io, postId): Main worker entry. Load post, call findComplementaryMatches + findWantedListMatches, group results by userId (one notification per user), send batched notifications via insertNotification + sendPushToUser + Socket.IO emit 'post-match'.
       - Export insertNotification and sendPushToUser as module-level helpers (extract from proposal.service.ts into a shared notification.helpers.ts file, or duplicate the pattern in post-match.service.ts to avoid coupling).

    2. Create post-match-worker.ts following match-worker.ts pattern exactly:
       - QUEUE_NAME = 'post-match'
       - queuePostMatch(postId, userId): Add job with 5-second delay and jobId `post-match:${postId}` for deduplication.
       - initPostMatchWorker(db, io): Create Worker that calls processPostMatch.
       - closePostMatchWorker(): Graceful shutdown.

    3. Wire into the system:
       - In posts.ts route: After successful createPost, call queuePostMatch(post.id, userId) to trigger background matching.
       - In server.ts: Import and call initPostMatchWorker(app.db, app.io) on startup, add closePostMatchWorker() to shutdown handler.

    4. Write tests for post-match.service.ts covering: complementary matching for both post types, wanted list proactive matching, self-exclusion, per-user notification batching.
  </action>

    - Proactive wanted list matching notifies users even without Seeking posts
    - BullMQ worker queues and processes post match jobs with 5-second delay
    - Socket.IO 'post-match' events emitted to affected users
    - Notifications batched per user to prevent spam
    - Worker registered in server.ts startup/shutdown
